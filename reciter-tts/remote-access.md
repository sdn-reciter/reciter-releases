# Доступ к своему TTS-серверу извне

Как открыть домашний TTS (XTTS/Supertonic/VoxCPM на портах 8002/8003/8004)
для телефона за пределами домашней сети. Ориентировано на вводные: **есть свой
зарубежный VPS с Xray и статичным IP + WARP**, роутер с маршрутизацией.

Общий обзор безопасного доступа (Tailscale/Caddy/Cloudflare) — в
[`self-hosted-server.md`](self-hosted-server.md). Здесь — про варианты именно
под свой VPS и ответ на вопрос про статический IP дома.

---

## TL;DR

- **Статический IP дома НЕ нужен.** Все рабочие варианты — «звонок наружу»
  (домашний ПК сам подключается к VPS/оверлею), а не входящее соединение к дому.
- **Лучший вариант — приватный оверлей (WireGuard/Tailscale).** Телефон видит
  ПК по внутреннему IP, наружу ничего не открывается, ключ поверх — как
  дополнительный слой.
- **Публичный HTTPS-эндпоинт на VPS (реверс-туннель)** — только если нужен
  стабильный адрес, которым можно делиться. Тогда обязательны TLS + API-ключ.
- Прямой проброс порта на роутере — крайний случай и **невозможен при CGNAT**.

---

## Нужен ли статический IP дома?

**Нет** — для вариантов 1–3 ниже. Домашний ПК инициирует исходящее соединение
к VPS/координатору, поэтому его адрес (даже «серый», за CGNAT) не важен.

Статический IP дома имел бы смысл только для «классического» проброса порта
(вариант 4), да и там его заменяет DDNS. А если провайдер даёт **CGNAT**
(частый случай мобильного/оптического интернета), проброс порта в принципе
невозможен — остаются варианты 1–3.

Что важно — это **статический IP у VPS** (он у тебя есть): к нему стабильно
«дозваниваются» и дом, и телефон.

### Как проверить CGNAT за 30 секунд

1. Посмотри WAN-IP в роутере (страница статуса).
2. Сравни с тем, что показывает `curl https://api.ipify.org` с домашнего ПК.
3. Если **не совпадают**, или WAN-IP из диапазонов `100.64.0.0/10`, `10.0.0.0/8`,
   `172.16–31.x`, `192.168.x` — ты за NAT провайдера (CGNAT). Проброс порта
   работать не будет, используй вариант 1 или 2.

## Сравнение вариантов

| Вариант | Статич. IP дома | Открытые порты наружу | Латентность | Когда выбирать |
|---|---|---|---|---|
| **1. WireGuard-хаб на своём VPS** | не нужен | нет (только UDP WG на VPS) | +RTT до VPS | нужен полный контроль, VPS уже есть |
| **2. Tailscale / Headscale** | не нужен | нет | низкая (часто P2P) | нужен минимум возни, лучшая латентность |
| **3. Реверс-туннель + HTTPS на VPS** | не нужен | 443 на VPS | +RTT до VPS | нужен публичный адрес/домен |
| **4. Проброс порта + DDNS** | желателен, но DDNS заменяет; **не при CGNAT** | да (443 дома) | минимальная | только если есть «белый» IP и хочется без VPS |

Про латентность: VPS зарубежный. Если телефон и дом в одной стране, трафик
через дальний VPS (варианты 1 и 3) делает «крюк» за границу. Вариант 2
(Tailscale) умеет пробивать NAT и соединять телефон↔дом **напрямую** (P2P), а
реле выбирает близкое — поэтому по скорости обычно выигрывает. Для чтения книг
это некритично: приложение предрендерит абзацы вперёд, разовый RTT прячется.

## Вариант 1 — WireGuard-хаб на своём VPS (рекомендую, если хочешь свой контроль)

Дом и телефон подключаются WireGuard-пирами к VPS. Телефон обращается к ПК по
внутреннему адресу (например `10.7.0.2:8004`). Наружу открыт только UDP-порт
WireGuard на VPS; TTS-порты не светятся нигде. WG живёт рядом с Xray на другом
порту — не мешают друг другу.

### 1.1. На VPS

```bash
sudo apt install wireguard
# ключи хаба
wg genkey | tee /etc/wireguard/priv | wg pubkey > /etc/wireguard/pub
```

`/etc/wireguard/wg0.conf`:

```ini
[Interface]
Address = 10.7.0.1/24
ListenPort = 51820
PrivateKey = <VPS_PRIV>
# маршрутизация между пирами (дом ↔ телефон через хаб)
PostUp   = sysctl -w net.ipv4.ip_forward=1
# домашний ПК
[Peer]
PublicKey = <HOME_PUB>
AllowedIPs = 10.7.0.2/32
# телефон
[Peer]
PublicKey = <PHONE_PUB>
AllowedIPs = 10.7.0.3/32
```

```bash
sudo systemctl enable --now wg-quick@wg0
# открой UDP 51820 в фаерволе VPS (и в WARP-конфиге, если исходящее идёт в WARP —
# слушающий сокет должен оставаться на реальном интерфейсе VPS, не в WARP).
```

### 1.2. На домашнем ПК (Linux)

```ini
[Interface]
Address = 10.7.0.2/24
PrivateKey = <HOME_PRIV>

[Peer]
PublicKey = <VPS_PUB>
Endpoint = <VPS_IP>:51820
AllowedIPs = 10.7.0.0/24
# держит туннель живым через NAT/CGNAT дома
PersistentKeepalive = 25
```

```bash
sudo systemctl enable --now wg-quick@wg0
```

Проверь, что TTS слушает на всех интерфейсах (в compose уже
`0.0.0.0:8004:8002`), тогда он доступен и по `10.7.0.2`.

### 1.3. На телефоне

Официальное приложение **WireGuard**. Конфиг:

```ini
[Interface]
Address = 10.7.0.3/24
PrivateKey = <PHONE_PRIV>

[Peer]
PublicKey = <VPS_PUB>
Endpoint = <VPS_IP>:51820
# СПЛИТ-ТУННЕЛЬ: через WG идёт только TTS-подсеть, остальной интернет — как обычно
AllowedIPs = 10.7.0.0/24
PersistentKeepalive = 25
```

В приложении Reciter: адрес сервера — `http://10.7.0.2:8004` (можно `http://` —
WireGuard уже шифрует; приложение разрешает cleartext). Ключ `API_KEY` — как
второй слой.

> Сплит-туннель (`AllowedIPs = 10.7.0.0/24`) важен: иначе весь трафик телефона
> пойдёт через дом/VPS. Нам нужно завернуть только обращения к TTS.

## Вариант 2 — Tailscale (проще всего) или Headscale (на своём VPS)

Ставишь Tailscale на домашний ПК и на телефон — оба попадают в одну приватную
сеть, телефон видит ПК по имени MagicDNS или адресу `100.x.y.z`. NAT пробивается
автоматически, часто соединение прямое (низкая латентность), реле — как запасной
путь. Наружу ничего не открывается, статический IP дома не нужен.

```bash
# домашний ПК
curl -fsSL https://tailscale.com/install.sh | sh && sudo tailscale up
# телефон — приложение Tailscale, вход тем же аккаунтом
```

В приложении Reciter: `http://<имя-ПК>.<tailnet>.ts.net:8004` или
`http://100.x.y.z:8004`.

Хочешь полный контроль без облака Tailscale — подними **Headscale** (свой
координатор) на том же VPS; клиенты те же, меняется только адрес координатора.
Это компромисс между простотой Tailscale и самостоятельностью варианта 1.

## Вариант 3 — Реверс-туннель + публичный HTTPS на VPS

Нужен, если хочется стабильный публичный адрес `https://tts.example.com`
(например, слушать без VPN-клиента на телефоне или дать доступ кому-то ещё).
Домашний ПК держит исходящий туннель к VPS, VPS терминирует TLS и проксирует в
дом. Статический IP дома не нужен.

Схема: `телефон → https://tts.example.com (Caddy на VPS) → туннель → дом:8004`.

- **Туннель дом→VPS:** проще всего `frp` (frpc дома → frps на VPS) или тот же
  WireGuard из варианта 1 (тогда Caddy проксирует на `10.7.0.2:8004`).
  Xray тоже умеет reverse (portal/bridge), но frp/WG прозрачнее в отладке.
- **TLS:** Caddy на VPS с авто-сертификатом Let's Encrypt (нужен домен, у тебя
  статический IP — A-запись на него).

`Caddyfile` на VPS:

```
tts.example.com {
    reverse_proxy 10.7.0.2:8004   # адрес дома в WG-оверлее (или frp-порт)
}
```

В приложении: `https://tts.example.com` + **обязательно** `API_KEY`.

> Это единственный вариант, где сервис реально виден в интернете. Поэтому:
> сильный ключ (`openssl rand -hex 32`), TLS, желательно rate-limit/fail2ban на
> VPS. Cleartext `http://` наружу — нельзя.

## Вариант 4 — Проброс порта на роутере + DDNS (не рекомендую)

Только если у тебя **«белый» публичный IP** (не CGNAT). Пробрасываешь на роутере
`443 → ПК`, поднимаешь Caddy для TLS, DDNS (например от роутера или Cloudflare)
следит за сменой домашнего IP — **статический IP не обязателен**. Минусы: сервис
открыт в интернет напрямую, больше поверхность атаки; при CGNAT не работает.
Если уже есть VPS — варианты 1–3 строго лучше.

---

## Замечание про WARP

WARP на VPS — это про **исходящий** трафик VPS (чистый egress-IP/обход
блокировок). На **входящий** доступ к дому он не влияет. Если маршрутизируешь
весь исходящий VPS в WARP, проследи, чтобы слушающие сокеты (WireGuard 51820,
Caddy 443) оставались на реальном интерфейсе VPS, иначе входящие подключения
сломаются. Обычно достаточно policy-routing’а: не заворачивать в WARP трафик
локальных портов-слушателей.

## Настройка приложения (любой вариант)

Dev-сборка: «Параметры → Чтение вслух → голос **Свой сервер** → плашка
**Свой TTS-сервер**»:

1. Адрес: `http://10.7.0.2:8004` (оверлей) или `https://tts.example.com` (публично).
2. Ключ доступа — значение `API_KEY` сервера.
3. Сохрани как пресет (XTTS / Supertonic / VoxCPM) — переключение между
   серверами без перенабора.

---

## Чеклист безопасности

- [ ] `API_KEY` задан на сервере всегда (даже поверх VPN — defence in depth).
- [ ] Предпочтение — оверлей (варианты 1/2): порты TTS не видны в интернете.
- [ ] Публичный доступ (3/4) — только по TLS и с ключом; cleartext наружу нет.
- [ ] `http://` без TLS — исключительно внутри шифрованного туннеля (WG/TS).
- [ ] На VPS открыт минимум портов (WG UDP или 443), фаервол по умолчанию deny.
- [ ] Публичный эндпоинт — rate-limit/fail2ban, по возможности allowlist IP.
