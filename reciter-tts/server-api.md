# Контракт TTS-сервера для Reciter

Что нужно реализовать, чтобы приложение подключилось к **вашему** серверу и
читало книги его голосом. Движок любой (XTTS, Piper, свой) — приложение общается
со всеми одинаково. Готовые примеры серверов — рядом (обзор в [`README.md`](README.md)).

## Минимум

Обязательны всего **два** эндпоинта:

| Эндпоинт | Зачем |
|---|---|
| `POST /tts` | принять текст → вернуть аудио |
| `GET /health` | проверка «жив ли сервер» |

Остальное (`/voices`, `/accent`) — опционально: без них просто не будет выбора
голоса и серверных ударений.

Подключение в приложении: «Параметры → Чтение вслух → движок **«Свой сервер»**»,
адрес `http://<IP>:<порт>` и (опц.) ключ.

## Как приложение шлёт запросы

- **Адрес** вводится в приложении: `http://host:port` или `https://host`. В
  локальной сети / через VPN можно `http`; наружу — только `https` или туннель.
- **Ключ.** Если в приложении задан ключ, каждый запрос получает заголовок
  `Authorization: Bearer <ключ>`. Ключ пуст → заголовка нет. Рекомендуется
  требовать ключ (даже за VPN) и отвечать `401` без него.
- **Один запрос = один абзац.** Приложение синтезирует по абзацу и
  предзагружает несколько вперёд, поэтому на одной GPU обрабатывайте запросы
  **по одному** (серию, а не параллельно), иначе гонка за память.
- **Скорость и высоту приложение НЕ передаёт** — накладывает их само при
  воспроизведении. Синтезируйте на нейтральной скорости, не «запекайте» темп в
  аудио (иначе применится дважды).
- Кодировка — **UTF-8**. Текст приходит уже нормализованным приложением.

## `POST /tts` (обязателен)

Тело — `multipart/form-data`:

| Поле | Обяз. | Описание |
|---|---|---|
| `text` | да | Абзац для озвучки. Может содержать метки ударений (см. ниже). |
| `language` | нет | ISO-код, по умолчанию `ru`. Берите часть до `-` (`ru-RU` → `ru`). |
| `voice` | нет | Имя голоса из `/voices`. Пусто → голос по умолчанию. |

**Ответ `200`** — тело аудио с заголовком `Content-Type: audio/wav` **или**
`audio/mpeg` (MP3). Частота/каналы на ваше усмотрение (обычно 24 кГц моно).
Приложение само определит формат и проиграет.

**Ошибка** — любой не-2xx, желательно телом `{"error": "..."}`. При ошибке или
недоступности приложение временно переключается на системный TTS и само
вернётся к вашему серверу, когда `/health` снова ответит.

## `GET /health` (обязателен)

Любой ответ `2xx` = «жив» (используется кнопкой «Тест» и фоновым опросом).
Отвечайте **быстро**. Полезно (не обязательно) вернуть JSON:

```json
{ "status": "ok", "server_version": "1", "engine": "my-tts", "voices": ["F1"] }
```

## Голоса (опционально)

| Эндпоинт | Тело / путь | Ответ |
|---|---|---|
| `GET /voices` | — | `{ "voices": ["F1","F2"], "default": "F1" }` |
| `POST /voices/upload` | `multipart`: `name` + `file` (wav/mp3/…) | `2xx` = добавлен |
| `DELETE /voices/{name}` | имя в пути | `2xx` = удалён |

Без `/voices` в приложении просто не будет выпадающего выбора голоса. Для
движков без клонирования (пресетные голоса) на `upload` верните понятную
ошибку JSON — приложение её покажет.

## Ударения — главный подводный камень (опционально)

От этого зависит, будет ли работать словарь ударений. Приложение может прислать
в `text` уже размеченный текст, а через `POST /accent` (тело `{"text":"…"}` →
`{"text":"…с ударениями…"}`) — расставить ударения нейромоделью RUAccent.

Форматы меток, которые может прислать клиент:

| Формат | Как выглядит | Пример |
|---|---|---|
| `mark` | акут `U+0301` **после** гласной | `за́мок` |
| `plus` | `+` **перед** ударной гласной | `з+амок` |
| `strip` | без меток | `замок` |

Что делать зависит от вашего движка:

- **Движок читает метки** (F5/ESpeech, XTTS с русским файнтюном на `+`) →
  **сохраняйте** метки в нужном движку формате. Тогда словарь реально управляет
  ударением.
- **Движок без G2P** (Supertonic, базовый XTTS, большинство end-to-end) →
  **срезайте** метки перед синтезом: для него `U+0301`/`+` — мусорный токен
  (в лучшем случае игнор, в худшем — запинка). Внешний словарь на таком движке
  не действует в принципе — это предел модели, не баг приложения.

Практично держать это переменной (`SYNTH_STRESS = strip|mark|plus`) и
**логировать финальный текст**, уходящий в модель.

## Минимальный сервер (FastAPI)

Хватает `POST /tts` и `GET /health`:

```python
from fastapi import FastAPI, Form, Header, HTTPException
from fastapi.responses import StreamingResponse, JSONResponse
from typing import Optional
import io, os

API_KEY = os.environ.get("API_KEY", "").strip()
app = FastAPI()

def require_auth(authorization: Optional[str]):
    if API_KEY and authorization != f"Bearer {API_KEY}":
        raise HTTPException(status_code=401, detail="Unauthorized")

@app.get("/health")
async def health():
    return {"status": "ok", "engine": "my-tts"}

@app.post("/tts")
async def tts(text: str = Form(...), language: str = Form("ru"),
              voice: Optional[str] = Form(None),
              authorization: Optional[str] = Header(None)):
    require_auth(authorization)
    lang = (language or "ru").split("-")[0].lower()
    try:
        # 1. при необходимости снять/нормализовать метки ударений
        # 2. синтез на НЕЙТРАЛЬНОЙ скорости -> WAV-байты
        wav_bytes: bytes = my_engine_synthesize(text, voice, lang)
        return StreamingResponse(io.BytesIO(wav_bytes), media_type="audio/wav")
    except Exception as e:
        return JSONResponse(status_code=500, content={"error": str(e)})
```

Добавив `/voices*` и `/accent`, включите выбор голосов и серверные ударения.
Полные рабочие реализации — в папках движков рядом.

## Чек-лист

- [ ] `POST /tts` принимает `multipart` (`text`, опц. `language`, `voice`) и
      отдаёт `audio/wav` или `audio/mpeg`.
- [ ] Синтез на **нейтральной скорости** (темп/питч накладывает приложение).
- [ ] Метки ударений обработаны под ваш движок (сохранены для stress-aware,
      срезаны для остальных).
- [ ] `GET /health` отвечает быстро и `2xx`.
- [ ] Если задан ключ — проверяется `Authorization: Bearer` на всех эндпоинтах.
- [ ] На одной GPU запросы обрабатываются по одному.
- [ ] (Опц.) `/voices`, `/voices/upload`, `DELETE /voices/{name}`, `/accent`.
