# Контракт своего TTS-сервера для Reciter

Единый **эталон протокола** между приложением и любым самостоятельным
TTS-сервером. Реализовав его, вы подключаете к Reciter свой движок (XTTS,
Supertonic, VoxCPM, ESpeech/F5, Piper, что угодно) — приложение общается со
всеми одинаково, отличается только начинка сервера.

Готовые серверы, собранные по этому контракту, — рядом (порты 8002–8007):
[`xtts-server`](xtts-server/README.md) · [`supertonic-server`](supertonic-server/README.md) ·
[`voxcpm-server`](voxcpm-server/README.md) · [`espeech-server`](espeech-server/README.md) ·
[`chatterbox-server`](chatterbox-server/README.md) · [`cosyvoice-server`](cosyvoice-server/README.md).
Обзор, запуск и доступ снаружи (Tailscale) — в [`README.md`](README.md).

> Функция «Свой сервер» доступна во **всех сборках** (dev и prod). В
> приложении: «Параметры → Чтение вслух → движок “Свой сервер”», адрес и
> (опц.) ключ. Edge/Google TTS остаются только в dev.

## 1. Транспорт и общие правила

- **Базовый адрес** вводится в приложении: `http://host:port`, `https://host`
  или host:port без схемы (тогда подставляется `http://`). Хвостовой `/`
  отбрасывается. Внутри LAN допустим cleartext-HTTP (пропускают обе сборки);
  снаружи — только HTTPS или приватный туннель.
- **Авторизация.** Если в приложении задан ключ, **каждый** запрос получает
  заголовок `Authorization: Bearer <ключ>`. Ключ пуст → заголовка нет. Сервер
  сам решает, требовать ли ключ (рекомендуется — да, даже за VPN).
- **Одна единица работы — один абзац.** Приложение синтезирует по абзацу и
  предзагружает несколько вперёд, поэтому на одной GPU **сериализуйте** запросы
  (один синтез за раз), иначе параллельные абзацы устроят гонку за память.
- **Таймауты.** Клиент ждёт долго (профиль с щедрым read-timeout) — домашней
  GPU не грех потратить несколько секунд на абзац. Но `/health` должен отвечать
  быстро (используется для частого опроса восстановления).
- **Кодировка** — UTF-8 везде. Текст приходит уже нормализованным приложением
  (раскрытие сокращений/чисел по возможности на стороне клиента).

## 2. Эндпоинты

Обязательны только **`POST /tts`** и **`GET /health`**. Остальные —
опциональны (без них просто не будет выбора голосов / серверных ударений).

### `POST /tts` — синтез (обязателен)

`multipart/form-data`:

| Поле | Обяз. | Описание |
|---|---|---|
| `text` | да | Абзац для озвучки. Может уже содержать метки ударений (см. §3). |
| `language` | нет | ISO-код, по умолчанию `ru`. Сервер берёт часть до `-` в нижнем регистре (`ru-RU` → `ru`). |
| `voice` | нет | Имя голоса из `/voices`. Пусто → голос сервера по умолчанию. |

**Ответ 200** — тело аудио: `audio/wav` **или** `audio/mpeg` (MP3). Приложение
стримит байты в файл и проигрывает через `MediaPlayer`, формат определяется
автоматически. Частота/каналы — на усмотрение сервера (обычно 24 кГц моно).

**Ошибка** — любой не-2xx; желательно телом JSON `{"error": "..."}` (первые
~300 символов уходят в лог приложения). При ошибке/недоступности приложение
временно откатывается на системный TTS и само возвращается на сервер, когда
`/health` снова отвечает.

> Приложение **не** передаёт скорость и высоту в `/tts`. Синтезируйте на
> нейтральной скорости — темп и питч приложение накладывает само при
> воспроизведении (`MediaPlayer.playbackParams`). Не «запекайте» скорость в
> аудио, иначе она применится дважды.

### `GET /health` — проверка живости (обязателен)

Любой ответ 2xx = «жив». Используется кнопкой «Тест» и фоновым опросом
восстановления. Полезно (не обязательно) вернуть JSON с диагностикой:

```json
{ "status": "ok", "server_version": "...", "engine": "...",
  "ruaccent": true, "voices": ["F1","F2"], "default_voice": "F1" }
```

Поле версии в логе старта и в `/health` — лучший способ убедиться, что
пересобрался свежий код, а не старый образ.

### `GET /voices` — список голосов (опц.)

```json
{ "voices": ["F1", "F2", "ilya"], "default": "F1" }
```

Пустой список/404 → в приложении просто не будет выпадающего выбора голоса.

### `POST /voices/upload` — добавить голос (опц.)

`multipart/form-data`: `name` (строка) + `file` (аудио: wav/mp3/m4a/ogg/flac).
Для движков с клонированием по образцу — эталонный wav; для пресетных
(Supertonic) — вернуть понятную ошибку JSON, приложение её покажет. Ответ 2xx =
успех.

### `DELETE /voices/{name}` — удалить голос (опц.)

Имя в пути URL-кодируется. Ответ 2xx = удалено. Пресеты можно не удалять.

### `POST /accent` — простановка ударений (опц.)

Нужен только для режима ударений **«Сервер»**. Тело JSON `{"text": "..."}`,
ответ JSON `{"text": "...с ударениями..."}`. Обычно за этим стоит нейромодель
RUAccent (точнее словаря на омографах: за́мок/замо́к по контексту). Ответы
кэшируются на клиенте. Формат меток — см. §3.

## 3. Ударения — что приходит в `text` и что с этим делать

Ключевой момент, от которого зависит, будет ли работать словарь ударений.
Режим ударений выбирается в приложении («Параметры → Чтение вслух →
Ударения»): `Словарь` (офлайн, в APK) · `Сервер` (`/accent`) · `Выкл`. Формат
метки тоже настраивается.

**Что доходит до `/tts`:**

- В режиме **«Словарь»** приложение расставляет ударения на устройстве и шлёт в
  `text` **уже размеченный** текст.
- В режиме **«Сервер»** приложение сначала зовёт `/accent`, потом шлёт
  результат в `text`.
- В режиме **«Выкл»** — текст без меток.

Форматы меток, которые может прислать клиент (и которые сервер должен уметь
принять/нормализовать):

| Формат | Как выглядит | Пример |
|---|---|---|
| `mark` | комбинируемый акут `U+0301` **после** гласной | `за́мок` |
| `plus` | `+` **перед** ударной гласной | `з+амок` |
| `strip` | без меток | `замок` |

**Главное правило для автора сервера — знать, stress-aware ли ваш движок:**

- **Движок читает метки** (F5-TTS/ESpeech, XTTS с русским файнтюном на
  `+`-разметке) → **сохраняйте** метки, приведя к нужному движку формату. Тогда
  словарь реально управляет ударением, включая омографы.
- **Движок работает по сырым символам без G2P** (Supertonic, базовый XTTS,
  большинство end-to-end моделей) → **срезайте** метки перед синтезом. Такой
  движок ставит ударение сам и не имеет входа для меток; `U+0301`/`+` для него —
  мусорный токен (в лучшем случае игнор, в худшем — запинка/лишний звук).
  Внешний словарь на таком движке **не действует в принципе** — это не баг
  приложения, а предел модели.

Практично держать это отдельной ручкой окружения (например,
`*_SYNTH_STRESS = strip|mark|plus`) и **логировать финальный текст**, уходящий
в модель, — тогда видно, применились метки или срезаны. Пример разбора для
конкретных движков: [`supertonic-server`](supertonic-server/README.md) (G2P-free,
метки не действуют) и [`espeech-server`](espeech-server/README.md) (stress-aware).

## 4. Минимальный сервер (референс)

Достаточно `POST /tts` и `GET /health`. Каркас на FastAPI:

```python
from fastapi import FastAPI, Form, Header, HTTPException
from fastapi.responses import StreamingResponse, JSONResponse
from typing import Optional
import io, os

API_KEY = os.environ.get("API_KEY", "").strip()
app = FastAPI()

def require_auth(authorization: Optional[str]):
    if not API_KEY:
        return
    if authorization != f"Bearer {API_KEY}":
        raise HTTPException(status_code=401, detail="Unauthorized")

@app.get("/health")
async def health():
    return {"status": "ok", "engine": "my-tts"}

@app.post("/tts")
async def tts(text: str = Form(...), language: str = Form("ru"),
              voice: Optional[str] = Form(None),
              authorization: Optional[str] = Header(None)):
    require_auth(authorization)
    lang = (language or "ru").split("-")[0].lower()
    try:
        # 1. при необходимости снять/нормализовать метки ударений (§3)
        # 2. синтез на НЕЙТРАЛЬНОЙ скорости -> PCM/WAV
        wav_bytes: bytes = my_engine_synthesize(text, voice, lang)
        return StreamingResponse(io.BytesIO(wav_bytes), media_type="audio/wav")
    except Exception as e:
        return JSONResponse(status_code=500, content={"error": str(e)})
```

Добавив `GET /voices`, `POST /voices/upload`, `DELETE /voices/{name}` и
`POST /accent`, вы включите выбор голосов и серверные ударения. Полные рабочие
реализации — в папках движков рядом.

## 5. Чек-лист совместимости

- [ ] `POST /tts` принимает `multipart` (`text`, опц. `language`, `voice`) и
      отдаёт `audio/wav` или `audio/mpeg`.
- [ ] Есть **голос по умолчанию** — образец в каждом запросе не нужен.
- [ ] Синтез на **нейтральной скорости** (темп/питч накладывает приложение).
- [ ] Метки ударений из `text` корректно обработаны под ваш движок (сохранены
      для stress-aware, срезаны для G2P-free) — §3.
- [ ] `GET /health` отвечает быстро и 2xx.
- [ ] Если задан ключ — проверяется `Authorization: Bearer` на всех эндпоинтах.
- [ ] На одной GPU запросы сериализованы.
- [ ] В логе старта — строка версии; финальный текст синтеза логируется.
- [ ] (Опц.) `/voices`, `/voices/upload`, `DELETE /voices/{name}`, `/accent`.
