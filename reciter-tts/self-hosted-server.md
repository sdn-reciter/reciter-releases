# Свой TTS-сервер (XTTS) для Reciter — безопасный доступ и подключение

XTTS-v2 на домашнем ПК с GPU: как **безопасно** слушать книги через интернет
(один пользователь — владелец) и как подключить приложение. Сначала тест по
Wi-Fi, потом интернет.

> Пишете **свой** сервер под другой движок? Общий контракт приложения
> (эндпоинты, форматы, ударения) — в [`server-api.md`](server-api.md). Этот
> файл — про конкретный XTTS-сервер и безопасный доступ снаружи.

> TL;DR
> 1. Пересобери сервер с **API-ключом** и **дефолтным голосом** (файлы в
>    `docs/reciter-tts/xtts-server/`) — приложению не придётся слать образец голоса в
>    каждом запросе.
> 2. Для интернета **не пробрасывай порт 8002 напрямую**. Самое простое и
>    безопасное — **Tailscale** (частный VPN, ничего не открывается наружу).
>    Альтернатива — **HTTPS через Caddy/Cloudflare Tunnel + ключ**.
> 3. В приложении (любая сборка): «Параметры → Чтение вслух → движок
>    “Свой сервер”», впиши адрес и ключ.

---

## 0. Модель угроз (почему так)

- Слушаешь только ты → нужен **один секрет** (API-ключ), а не система аккаунтов.
- Сервер стоит на машине с GPU и работает в Docker от root → если его
  «ломанут», получат доступ ко всей машине. Поэтому цель — **не светить порт
  в интернет** и требовать ключ.
- XTTS принимает `speaker_wav` и умеет **клонировать любой голос** — ещё одна
  причина закрыть доступ: чужой не должен слать туда произвольные запросы.

Три правила:
1. Никакого «проброса 8002 на роутере в мир» без TLS и ключа.
2. Трафик наружу — только шифрованный (VPN-туннель или HTTPS).
3. Всегда — секретный ключ, даже поверх VPN (defence in depth).

## 1. Разверни сервер (ключ + дефолтный голос)

Сервер хранит эталонные голоса у себя (`voices/`), клиент шлёт только текст:

- **`DEFAULT_SPEAKER`** — имя голоса по умолчанию из `voices/`.
- **`API_KEY`** — если задан, каждый запрос обязан прислать
  `Authorization: Bearer <key>`; иначе `401`.
- Пустой `API_KEY` = без авторизации (**только** для изолированной Wi-Fi-сети).

Готовый комплект — в `docs/reciter-tts/xtts-server/` (структура и быстрый старт описаны в
[`xtts-server/README.md`](xtts-server/README.md)):
`Dockerfile`, `server.py`, `docker-compose.yml`, `.env.example`, `Caddyfile`,
папка `voices/` для референсов голосов.

### Шаги

```bash
# 1. Скопируй комплект docs/reciter-tts/xtts-server/ на машину с GPU
cd xtts-server

# 2. Ключ и настройки
cp .env.example .env
echo "XTTS_API_KEY=$(openssl rand -hex 32)" >> .env   # значение понадобится в приложении

# 3. Дефолтный голос (эталон для клонирования)
mkdir -p voices
cp /путь/к/образцу.wav voices/default.wav   # 6–20 сек чистой речи

# 4. Собери и запусти
docker compose up -d --build
docker compose logs -f | grep -E "API auth|Voices dir|RUAccent|Model loaded"
```

`Dockerfile` уже включает установку RUAccent (ударения); если встраиваешь в
свой Dockerfile — фрагмент в `Dockerfile.snippet`. `ffmpeg` нужен для приёма
голосов в любом формате (в комплекте он есть).

### Обновление сервера (важно!)

`docker compose up -d --build` пересобирает образ из **локальных** файлов. Если
ты скопировал `server.py` вручную и правил комплект в репозитории — сначала
**обнови файлы на машине с сервером**, иначе пересоберётся старый код:

```bash
# вариант А: репозиторий склонирован на ПК
git pull

# вариант Б: копировал вручную — перекопируй свежие файлы
cp /путь/к/docs/reciter-tts/xtts-server/{server.py,docker-compose.yml,.env.example} .

docker compose up -d --build
```

**Проверь, что крутится свежий код** — в логе при старте есть строка версии:

```bash
docker compose logs | grep SERVER_VERSION
# Reciter XTTS server — SERVER_VERSION 6 (…)
curl -s http://localhost:8002/health | python3 -m json.tool   # поле "server_version"
```

Если версия ниже ожидаемой или строки нет — значит образ собран из старого
`server.py`: обнови файл и пересобери. Именно это чаще всего выглядит как
«ничего не изменилось».

Проверь дефолтный голос без загрузки образца:

```bash
# Внутри LAN, с ключом:
curl -X POST http://localhost:8002/tts \
  -H "Authorization: Bearer <ТВОЙ_КЛЮЧ>" \
  -F "text=Проверка синтеза без загрузки голоса." \
  -F "language=ru" --output test.wav && echo OK
```

> Примечание про порт в `docker-compose.yml`: по умолчанию он теперь
> `127.0.0.1:8002:8002` — сервер виден **только на самой машине** (правильно для
> работы за прокси/VPN). Для **теста по Wi-Fi** временно поставь `"8002:8002"`,
> чтобы телефон в той же сети достучался до `http://<IP-ПК>:8002`.

## 2. Тест по Wi-Fi (первый шаг)

1. В `docker-compose.yml` временно выставь `ports: ["8002:8002"]`, `up -d`.
2. Узнай IP ПК в сети: `ip addr | grep 192.168` (например `192.168.1.50`).
3. Телефон — в той же Wi-Fi-сети.
4. В приложении: «Параметры → Чтение вслух → движок
   **“Свой сервер”**». Впиши:
   - **Адрес сервера:** `http://192.168.1.50:8002`
   - **Ключ доступа:** значение `XTTS_API_KEY` (или пусто, если не задал).
5. Нажми «Тест голоса». Пошёл звук — работает.

> На Wi-Fi это обычный HTTP (без шифрования), поэтому и «только для теста в
> доверенной сети». Cleartext-HTTP пропускают обе сборки (dev и prod), но
> доверие ограничено системными CA — для интернета всё равно нужен туннель
> или HTTPS (см. §3).

## 2a. Не подключается с телефона? Диагностика

Симптомы из реального лога: приложение получало `HTTP 503`, затем `timeout` —
т.е. телефон **не доставал** до сервера. Проверь по порядку:

1. **Сервер слушает в LAN, а не только localhost.** В новом compose порт по
   умолчанию `0.0.0.0` (виден в сети). Проверь:
   ```bash
   docker ps        # должно быть  0.0.0.0:8002->8002/tcp  (а НЕ 127.0.0.1:8002)
   ```
   Если видишь `127.0.0.1:8002` — в `.env` поставь `XTTS_BIND=0.0.0.0` и
   `docker compose up -d`.

2. **Дождись загрузки модели.** Первые ~1–2 минуты после `up` сервер грузит
   XTTS — ранние запросы падают. Готовность:
   ```bash
   docker compose logs -f | grep "Model loaded"
   ```

3. **Проверь с САМОГО ТЕЛЕФОНА** (тот же Wi-Fi), открой в браузере телефона:
   ```
   http://<IP-ПК>:8002/health
   ```
   IP компа: `hostname -I` (напр. `192.168.1.50`). Если `/health` не
   открывается в браузере телефона — проблема сетевая (пункты 4–5), приложение
   ни при чём.

4. **Файрвол ПК.** Открой порт:
   ```bash
   sudo ufw allow 8002/tcp     # или временно: sudo ufw disable
   ```

5. **Правильный адрес в приложении.** Именно `http://<IP-ПК-в-Wi-Fi>:8002` —
   НЕ `127.0.0.1`, НЕ адрес контейнера (`172.x`), НЕ `localhost`. Ключ пустой,
   если `API_KEY` не задан.

6. **Движок «Свой сервер» есть в любой сборке** (dev и prod), включая запись
   эталона голоса с микрофона и загрузку файлом. RECORD_AUDIO запрашивается
   в рантайме по нажатию «Записать».

> `172.18.0.1` в логах сервера — это Docker-шлюз (твой `curl` с самого ПК), а не
> телефон. То, что curl с ПК работает, ещё не значит, что виден из сети — смотри
> пункт 1.

## 3. Интернет — безопасно (выбери один вариант)

> Есть свой VPS (Xray/статичный IP) и вопрос про статический IP дома? Разбор
> вариантов именно под свой сервер — в [`remote-access.md`](remote-access.md)
> (WireGuard-хаб, Headscale, реверс-туннель с публичным HTTPS, про CGNAT/WARP).

### Вариант A — Tailscale (рекомендую: проще и безопаснее всего)

WireGuard-VPN без открытия портов. Телефон и ПК попадают в одну частную сеть,
трафик шифруется, порт в интернет **не светится вообще**.

```bash
# на ПК с сервером
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
tailscale ip -4        # частный адрес, напр. 100.101.102.103
```

- Поставь **Tailscale** на телефон, войди тем же аккаунтом.
- В `docker-compose.yml` оставь порт `127.0.0.1:8002:8002` **или** забинди на
  tailscale-интерфейс: `"100.101.102.103:8002:8002"`.
- В приложении адрес: `http://100.101.102.103:8002`, ключ — как обычно.
- Плюсы: ничего не открыто наружу, шифрование «из коробки», работает из любой
  сети. Минус: нужен клиент Tailscale на телефоне (бесплатно для личного).

> Даже внутри Tailscale **оставь API-ключ** — если телефон потеряешь, ключ
> можно сменить одной пересборкой, не трогая VPN.

### Вариант B — Cloudflare Tunnel (HTTPS-хостнейм, без открытых портов)

```bash
# на ПК: cloudflared туннель на локальный сервис
cloudflared tunnel --url http://localhost:8002
```

Даёт HTTPS-URL (`https://<random>.trycloudflare.com` для быстрого теста, либо
именованный туннель на своём домене для постоянного). Порты на роутере не
открываются — соединение исходящее. В приложении укажи `https://…` и ключ.
Для постоянного использования настрой именованный туннель + (желательно)
Cloudflare Access.

### Вариант C — HTTPS через Caddy + проброс порта (если нужен свой домен)

Только если реально нужен публичный HTTPS-эндпоинт. Требует домен/DDNS и
проброс **443** (не 8002!) на машину с Caddy.

1. Сервер бинди на `127.0.0.1:8002` (как в compose по умолчанию).
2. `Caddyfile` (в `docs/reciter-tts/xtts-server/`) — подставь свой домен и ключ. Caddy сам
   получит и обновит TLS-сертификат и отсечёт запросы без `Authorization`.
3. На роутере: `443 → машина-с-Caddy:443`. Порт 8002 наружу **не** пробрасывай.
4. В приложении: `https://tts.твой-домен` + ключ.

### Чего НЕ делать

- ❌ Проброс `8002` с роутера в интернет «как есть».
- ❌ Пустой `API_KEY` при доступе снаружи.
- ❌ Самоподписанный сертификат на публичном хосте (приложение доверяет только
  системным CA — соединение не установится).

## 4. Подключение в приложении

Движок доступен **во всех сборках** (dev и prod), включая запись эталона голоса
с микрофона и загрузку файлом (RECORD_AUDIO запрашивается в рантайме по
нажатию «Записать»). Edge/Google TTS по-прежнему только dev.

- «Параметры → Чтение вслух» → в выпадающем списке движков выбери
  **«Свой сервер»**.
- Появятся поля:
  - **Адрес сервера** — `http://<ip>:8002`, `http://100.x.x.x:8002`
    (Tailscale) или `https://tts.домен` (Caddy/Cloudflare).
  - **Ключ доступа** — значение `XTTS_API_KEY` (можно пусто на доверенной Wi-Fi).
- «Тест голоса» проверит связь. Дальше книги читаются этим голосом; приложение
  синтезирует по одному абзацу и подкачивает следующие заранее
  (настройка «предзагрузка»), так что паузы между абзацами минимальны.

Технически приложение шлёт `POST {адрес}/tts` с полями `text` и `language` и
заголовком `Authorization: Bearer <ключ>` (если ключ задан), получает WAV и
проигрывает его. Пословной подсветки (караоке) для своего сервера нет —
используется послоговая оценка, как для Google TTS.

## 4a. Клонирование голоса (свои голоса)

Сервер хранит эталонные образцы в `voices/` и клонирует по ним. Управлять
можно **прямо из приложения** (dev): движок «Свой сервер» → блок
**«Голоса для клонирования»**:

- **Активный голос** — список голосов с сервера + «По умолчанию (сервер)».
  Выбор радиокнопкой; корзина — удалить.
- **Новый голос** — «Записать» (микрофон, 6–20 сек чистой речи) или «Выбрать
  файл», затем имя и «Загрузить голос». Приложение шлёт запись на сервер,
  где `ffmpeg` перекодирует её в WAV 24 кГц моно.

Через curl это выглядит так:

```bash
# загрузить свой образец как голос "ilya"
curl -X POST http://localhost:8002/voices/upload \
  -H "Authorization: Bearer <КЛЮЧ>" \
  -F "name=ilya" -F "file=@978_hvostikov-ilya_demo.mp3"

# список голосов / удалить
curl -H "Authorization: Bearer <КЛЮЧ>" http://localhost:8002/voices
curl -X DELETE -H "Authorization: Bearer <КЛЮЧ>" http://localhost:8002/voices/ilya

# синтез конкретным голосом
curl -X POST http://localhost:8002/tts -H "Authorization: Bearer <КЛЮЧ>" \
  -F "text=Проверка голоса." -F "voice=ilya" --output ilya.wav
```

Совет по качеству эталона: один говорящий, без музыки/шума/эха, ровная
интонация, 6–20 секунд. Слишком длинный или шумный образец ухудшает клон.

## 4b. Ударения — режим «Сервер» (нейро-RUAccent)

Режим ударений выбирается в «Параметры → Чтение вслух → **Ударения**»:
`Словарь` (по умолчанию, офлайн-словарь в APK — см.
[`accent-dictionary.md`](accent-dictionary.md)) · `Сервер` (dev) · `Выкл`.

Режим **«Сервер»** прогоняет текст через `POST /accent` этого сервера —
нейромодель RUAccent точнее словаря на **омографах** (за́мок/замо́к), потому
что смотрит на контекст.

- На сервере: `USE_RUACCENT=1` в compose (Dockerfile уже ставит RUAccent).
- Область действия: файловые движки — Edge, Google, свой сервер, «системный
  HQ». Обычный системный и браузерный движки остаются на офлайн-словаре: у
  них точная караоке-подсветка, которую сетевой шаг сломал бы.
- Ответы `/accent` кэшируются — на перемотках сервер не дёргается повторно.
- Сервер недоступен → откат к тексту без ударений (не падаем).

### Что XTTS реально умеет с ударениями (важно)

**Базовая XTTS-v2 меток ударения НЕ понимает — никаких.** Проверено по коду
Coqui TTS: клинер для `ru` не удаляет `U+0301`, а в BPE-словаре токенизатора
такого символа нет — каждый акут превращается в незнакомый токен, и модель
**запинается** ровно на ударениях. «+» в тексте она читает как мусор/паузу.
Ударения базовая модель ставит **сама из контекста** (обучена на русском
аудио) и в основном верно; ошибается на редких словах и омографах.

Поэтому форматы разделены на две независимые ручки:

| Переменная (.env) | Для чего | Значения |
|---|---|---|
| `XTTS_STRESS_MODE` | формат ответа `/accent` для **внешних** движков — Edge/Google/системный `U+0301` понимают | `mark` (по умолчанию) · `plus` · `strip` |
| `XTTS_SYNTH_STRESS` | что скармливать **самой XTTS** при синтезе | `strip` (по умолчанию — метки снимаются) · `plus` (для файнтюна, см. ниже) · `mark` (базовой модели противопоказано) |

Сервер нормализует любой входной текст `/tts` (сырой `+`, `U+0301`,
смешанный) к формату `XTTS_SYNTH_STRESS`, так что запинок от меток больше
не будет независимо от того, что прислал клиент.

### Управляемые ударения в XTTS: русский файнтюн

Единственный способ заставить XTTS **соблюдать** ударения — модель,
дообученная на размеченном тексте. Например,
[`xttsv2_banana`](https://huggingface.co/Ftfyhh/xttsv2_banana) — русский
файнтюн XTTS-v2, обученный на `+`-метках **перед** ударной гласной
(`молок+о`) — ровно формат RUAccent. Подключение:

1. Скачай с HF `model.pth`, `config.json`, `vocab.json` в папку `./model`
   рядом с `docker-compose.yml` (том уже прокинут).
2. В `.env`:
   ```
   XTTS_MODEL_DIR=/app/model
   XTTS_SYNTH_STRESS=plus
   ```
3. `docker compose up -d` — в логе появится `Custom model dir: /app/model`.

Тогда RUAccent-разметка начнёт реально управлять произношением, включая
омографы (за́мок/замо́к по контексту — это и есть нейромодель RUAccent).


## 5. Улучшения сервера (по желанию)

- **Прогрев/пул:** модель уже держится в памяти — хорошо. Для длинных книг
  включи на сервере ограничение параллелизма (одна GPU — один запрос за раз),
  чтобы предзагрузка нескольких абзацев не устраивала гонку за память.
- **Немного латентности:** можно отдавать `audio-24khz` MP3 вместо WAV, чтобы
  снизить трафик на мобильной сети (приложение проигрывает и то, и другое).
- **Не root:** запусти контейнер под непривилегированным пользователем
  (`user:` в compose) — снизит ущерб при компрометации.
- **Обновления:** периодически `docker compose build --pull` для свежих базовых
  образов CUDA/пакетов.
- **Логи:** сервер печатает `auth ON/OFF` и наличие дефолтного голоса при
  старте — удобно проверить конфиг после пересборки.

## 5a. «Странные звуки» в конце фраз (артефакты XTTS)

XTTS-v2 иногда доклеивает в конце короткой фразы посторонний звук — призвук,
«бормотание» или щелчок. Это известная особенность модели: GPT-декодер без
чёткого сигнала «стоп» добирает лишние токены. Сервер лечит это тремя
независимыми способами (все включены по умолчанию, ничего настраивать не
нужно):

1. **Финальная пунктуация.** Если фраза заканчивается буквой, сервер сам
   добавляет точку — это лучший сигнал модели «здесь конец».
2. **Параметры сэмплинга** против добора токенов (см. env ниже).
3. **Пост-обработка волны:** срез хвостовой тишины и короткого «выброса» после
   явной паузы + мягкий fade, чтобы срез не щёлкал.
4. **Посентенсный синтез.** Сервер сам делит абзац на предложения,
   синтезирует каждое отдельно, чистит хвост КАЖДОГО и склеивает через
   фиксированную паузу (`XTTS_SENT_PAUSE_MS`) — артефакты между предложениями
   внутри абзаца тоже убираются. Заодно эмбеддинг голоса кэшируется
   (раньше пересчитывался на каждый запрос) — синтез быстрее.

Дополнительно `/tts` санирует вход: если клиент прислал текст со старой
`+`-разметкой ударений (например, из устаревшего кэша `/accent`), сервер сам
конвертирует её в настроенный формат — «растянутая» речь от `+` в тексте
исключена.

Если хвост всё равно иногда слышен — поднимите штраф за повторы. Все ручки
задаются в `.env` (перезапуск: `docker compose up -d`):

| Переменная | По умолчанию | Что делает |
|---|---|---|
| `XTTS_REPETITION_PENALTY` | `6.0` | Главная ручка против «бормотания». Поднимите до `8–12`, если хвост слышен. |
| `XTTS_TEMPERATURE` | `0.7` | Ниже → стабильнее и меньше галлюцинаций (но монотоннее). |
| `XTTS_TOP_K` / `XTTS_TOP_P` | `50` / `0.85` | Сужение выборки токенов. |
| `XTTS_LENGTH_PENALTY` | `1.0` | Штраф за длину генерации. |
| `XTTS_TRIM_SILENCE` | `1` | Резать тишину в начале/конце. |
| `XTTS_TRIM_TAIL` | `1` | Отрезать короткий «выброс» после паузы в конце. |
| `XTTS_TAIL_MIN_GAP_MS` | `200` | Мин. пауза перед «выбросом», чтобы считать его артефактом. |
| `XTTS_TAIL_MAX_LEN_MS` | `250` | Макс. длина отрезаемого «выброса» (длиннее — не трогаем). |
| `XTTS_SILENCE_DB` | `-45` | Порог тишины относительно пика. |
| `XTTS_FADE_MS` | `12` | Длина fade-in/out на срезах. |
| `XTTS_SENT_PAUSE_MS` | `250` | Пауза между предложениями при посентенсной склейке. |

> Пост-обработка консервативна: настоящий короткий последний слог (после
> паузы < `XTTS_TAIL_MIN_GAP_MS`) не режется — отрезается только явно
> оторванный «выброс» в самом конце.
